## 2024-05-24 - [Avoid Redundant HTML Parsing in markdownify]
**Learning:** `markdownify()` function takes an HTML string. Passing a `BeautifulSoup` element to it requires serializing the tree back into a string (`str(html_element)`), only for `markdownify` to immediately re-parse it into a new `BeautifulSoup` tree under the hood.
**Action:** Use `MarkdownConverter(heading_style="ATX").convert_soup(html_element)` directly instead of `markdownify(str(html_element), heading_style="ATX")` to skip redundant string serialization and parsing.

## 2025-03-05 - BeautifulSoup Traversal Optimization
**Learning:** When searching for early-occurring document elements (like an author byline), the main performance hazard is switching from a short-circuiting `soup.find()` (or a small fallback chain of `find()` calls) to a combined `soup.find_all()` that must traverse the entire tree just to return the first match. `soup.find()` always stops at the first match, even when you use a regex for the attribute check, whereas `find_all()` forces a full-document traversal. A regex-based `find()` is slightly heavier per comparison than a simple equality check, but it still short-circuits. For deep elements like the main post body, where you effectively have to walk most of the document anyway, combining multiple class checks into a single regex `find()` (e.g. `soup.find(class_=re.compile(r"^(postBody|post-body|PostBody)$"))`) can be highly effective, reducing multiple near-full traversals to a single pass that BeautifulSoup can optimize internally.
**Action:** Prefer `find()` over `find_all()` whenever you only need the first match, keeping simple `find()` calls (or a short fallback chain) for elements known to appear early in the DOM. Only consolidate into a single regex `find()` when you are already committed to a deep or full traversal (such as locating the main content container), and avoid using `find_all()` solely to emulate a combined search in those cases.
